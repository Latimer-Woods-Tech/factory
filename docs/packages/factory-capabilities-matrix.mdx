## Detailed Package Descriptions

### Foundation Tier (Stable, Zero-Risk)

#### **@latimer-woods-tech/errors**
- **What it does:** Defines standard error types for all Factory apps
- **Key exports:** `FactoryBaseError`, `FactoryErrorShape`, `ErrorCodes`, error subclasses (`AuthError`, `NotFoundError`, `ValidationError`, `InternalError`, `ForbiddenError`)
- **When to use:** Everywhere — catch Factory errors, throw errors that consumers can handle uniformly
- **Config needed:** None
- **Dependencies:** None
- **Videoking usage:** All error handling; DLQ failure recording; webhook error classification
- **Real example:**
  ```typescript
  import { AuthError, ValidationError, ErrorCodes } from '@latimer-woods-tech/errors';
  
  if (!email) {
    throw new ValidationError('Email is required', {
      code: ErrorCodes.VALIDATION_ERROR,
      context: { field: 'email' }
    });
  }
  ```

#### **@latimer-woods-tech/monitoring**
- **What it does:** Sentry error tracking + APM (Application Performance Monitoring)
- **Key exports:** `initSentry`, `captureException`, `captureMessage`, error middleware
- **When to use:** Initialize once in Worker entry point; all unhandled errors auto-captured
- **Config needed:** `SENTRY_DSN`, environment name (dev/staging/prod)
- **Dependencies:** errors
- **Videoking usage:** Auto-capture of webhook failures, API errors, DLQ events; performance traces for video render pipeline
- **Real example:**
  ```typescript
  import { initSentry } from '@latimer-woods-tech/monitoring';
  
  export default initSentry({
    dsn: env.SENTRY_DSN,
    environment: env.ENVIRONMENT,
    release: env.WORKER_VERSION,
    tracesSampleRate: 0.1,
  })(app);
  ```

#### **@latimer-woods-tech/logger**
- **What it does:** JSON structured logging to stdout
- **Key exports:** `createLogger`, logger with `.info()`, `.warn()`, `.error()`, `.debug()`
- **When to use:** Log meaningful events (job started, payment processed, error occurred)
- **Config needed:** None (JSON output auto-formatted)
- **Dependencies:** errors, monitoring
- **Videoking usage:** Payout batch logs, webhook processing, admin API activity
- **Real example:**

  import { createLogger } from '@latimer-woods-tech/logger';

### Active Tier (Production Use, Well-Tested)

#### **@latimer-woods-tech/auth**
- **What it does:** JWT token creation, validation, RBAC (role-based access control), middleware
- **Key exports:** `createToken`, `verifyToken`, `rbacMiddleware`, token types
- **When to use:** Every protected API endpoint; all routes requiring authorization
- **Config needed:** `JWT_SECRET`, token expiry constants
- **Dependencies:** errors, logger
- **Videoking usage:**
  - Auth middleware on all `/api/*` routes
  - RBAC for admin endpoints (DLQ recovery requires `admin` role)
  - Payout batch creation restricted to `admin`
  - Creator routes require `creator` or `admin` role
- **Token structure:**
  ```typescript
  interface TokenPayload {
    sub: string;          // user ID
    tenantId: string;     // org/workspace ID
    role: 'owner' | 'admin' | 'member' | 'viewer';
    iat: number;          // issued at
    exp: number;          // expires at
  }
  ```
- **Real example:**
  ```typescript
  import { rbacMiddleware } from '@latimer-woods-tech/auth';
  
  app.post('/api/admin/payouts/batch', 
    rbacMiddleware(['admin', 'owner']),
    async (c) => {
      // POST body: { startDate, endDate }
      // Only admin/owner can create payout batches
    }
  );
  ```

#### **@latimer-woods-tech/neon**
- **What it does:** Drizzle ORM client bound to Neon Hyperdrive, migrations layer
- **Key exports:** `createDb`, `setTenantId`, `runMigrations`, re-exported `sql` from drizzle-orm
- **When to use:** All database access; initialization in Worker entry point
- **Config needed:** `env.DB` Hyperdrive binding
- **Dependencies:** errors, logger
- **Videoking usage:**
  - All 92-table schema queries
  - Row-level security (RLS) per tenant via `setTenantId()`
  - Migrations for schema changes
  - DLQ, payout_batches, video_calendar tables
- **Real example:**
  ```typescript
  import { createDb, setTenantId, sql } from '@latimer-woods-tech/neon';
  
  const db = createDb(env.DB);
  
  // Set tenant context for RLS
  await setTenantId(db, tenantId, async () => {
    const creators = await db.query.creators.findMany();
    // Query automatically filtered by current tenant
  });
  ```

#### **@latimer-woods-tech/stripe**
- **What it does:** Stripe client, webhook validation, subscription lifecycle helpers
- **Key exports:** `createStripeClient`, `stripeWebhookHandler`, `SubscriptionStatus`, methods for checkout, cancellation
- **When to use:** Payment endpoints; webhook handlers; subscription state machine
- **Config needed:** `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `STRIPE_PUBLISH_KEY`
- **Dependencies:** errors, logger, neon
- **Videoking usage:**
  - Viewer subscriptions: Citizen tier (free), VIP tier (premium)
  - Creator payouts via Stripe Connect transfers
  - Webhook handlers for charge.transfer, customer.subscription.updated
  - Payout batch execution sends Stripe API calls to each creator's connected account
- **Subscription states:**
  ```typescript
  type SubscriptionStatus = 
    | 'active'        // Subscription is active
    | 'trialing'      // In free trial
    | 'past_due'      // Payment failed, retrying
    | 'canceled'      // Canceled by customer or system
    | 'none';         // No subscription (free tier)
  ```
- **Real example:**
  ```typescript
  import { createStripeClient, stripeWebhookHandler } from '@latimer-woods-tech/stripe';
  
  const stripe = createStripeClient(env.STRIPE_SECRET_KEY);
  
  app.post('/webhooks/stripe', stripeWebhookHandler({
    webhookSecret: env.STRIPE_WEBHOOK_SECRET,
    stripeClient: stripe,
    handlers: {
      'upgraded': async (status) => {
        // Update user subscription tier
      },
      'canceled': async (status) => {
        // Handle cancellation
      },
    },
  }));
  ```

#### **@latimer-woods-tech/llm**
- **What it does:** Multi-provider LLM chain: tries Anthropic Claude first, falls back to Grok, then Groq
- **Key exports:** `createLlmChain`, `chatCompletion`, `streamCompletion`, provider fallback logic
- **When to use:** Any LLM inference (content generation, script writing, embeddings)
- **Config needed:** `ANTHROPIC_API_KEY`, `GROK_API_KEY`, `GROQ_API_KEY`
- **Dependencies:** errors, logger
- **Videoking usage:**
  - Script generation for video rendering (via render-video.yml GitHub Action)
  - Suggested copy for creator profiles
  - Video title/description optimization
  - Fallback: if Anthropic is rate-limited, automatically retry with Grok
- **Real example:**
  ```typescript
  import { createLlmChain } from '@latimer-woods-tech/llm';
  
  const llm = createLlmChain({
    anthropicKey: env.ANTHROPIC_API_KEY,
    grokKey: env.GROK_API_KEY,
    groqKey: env.GROQ_API_KEY,
  });
  
  const script = await llm.chatCompletion({
    model: 'claude-3-5-sonnet',
    messages: [
      { role: 'user', content: 'Write a script about ${topic}' }
    ],
  });
  // Automatically retries with Grok or Groq if Anthropic fails
  ```

#### **@latimer-woods-tech/analytics**
- **What it does:** Dual-write instrumentation to PostHog (product analytics) and factory_events table (revenue + compliance events)
- **Key exports:** `initAnalytics`, `.track()`, `.identify()`, `.businessEvent()`, `.page()`
- **When to use:** Every meaningful user action (signup, watch, subscribe, create content)
- **Config needed:** `POSTHOG_API_KEY`
- **Dependencies:** errors, neon
- **Videoking usage:**
  - Viewer engagement funnels: landing → signup → first watch → subscribe → renew
  - Creator retention funnels: publish → get views → earnings
  - Revenue events: subscription.created, subscription.renewed, payout.success
  - Compliance events: gdpr.data_request, account.deleted
  - Aligns with journeys from Phase A ("user journeys & telemetry")
- **Real example:**
  ```typescript
  import { initAnalytics } from '@latimer-woods-tech/analytics';
  
  const analytics = initAnalytics({
    postHogKey: env.POSTHOG_API_KEY,
    db: env.DB,
    appId: 'videoking',
  });
  
  // Product event (both PostHog + factory_events)
  await analytics.track('video.watched', {
    videoId: video.id,
    duration: watchDuration,
    platform: 'web',
  }, viewerId);
  
  // Revenue event (factory_events only, never sent to PostHog)
  await analytics.businessEvent('revenue.subscription_created', {
    customerId: customer.id,
    tier: 'vip',
    amount: 9.99,
    currency: 'USD',
  }, viewerId);
  ```

#### **@latimer-woods-tech/schedule**
- **What it does:** Video production calendar + priority scoring based on PostHog engagement signals
- **Key exports:** `VideoCalendarService`, `.getPendingJobs()`, `.scorePriority()`, `.updateJobStatus()`
- **When to use:** In cron Workers to dispatch video renders; in admin UI to track pipeline
- **Config needed:** Database, PostHog access
- **Dependencies:** errors, neon, video
- **Videoking usage:**
  - Daily cron fires; queries pending videos from `video_calendar` table
  - Scores engagement (from PostHog) to prioritize render queue
  - Dispatches GitHub Actions workflow_dispatch for highest-priority videos
  - Updates job status as renders complete
- **Real example:**
  ```typescript
  import { createScheduleService } from '@latimer-woods-tech/schedule';
  
  const schedule = createScheduleService(db);
  
  // In cron Worker
  const pending = await schedule.getPendingJobs();
  const scored = await Promise.all(
    pending.map(job => schedule.scorePriority(job))
  );
  
  const topJobs = scored
    .sort((a, b) => b.score - a.score)
    .slice(0, 5);
  
  for (const job of topJobs) {
    await dispatch(`github-actions`, 'render-video', {
      videoCalendarId: job.id,
      topic: job.topic,
    });
  }
  ```

## Decision Tree: Build vs. Consume

```
START: Do I need feature X?
  │
  ├─ Does Factory have a package for it?
  │   │
  │   ├─ YES (found package)
  │   │   │
  │   │   ├─ Is it in Foundation or Active tier?
  │   │   │   │
  │   │   │   ├─ YES → **CONSUME IT** (production-ready, zero risk)
  │   │   │   │
  │   │   │   └─ NO (Staging) → Can I wait 1–2 sprints?
  │   │   │       │
  │   │   │       ├─ YES → **PLAN TO CONSUME** (mark in tech debt; milestone = Phase C)
  │   │   │       │
  │   │   │       └─ NO → **CONSIDER BUILDING** (but evaluate if you can rework workflow)
  │   │   │
  │   │   └─ Does package's API meet 90% of my needs?
  │   │       │
  │   │       ├─ YES → **CONSUME IT** + (optional) request minor API extension
  │   │       │
  │   │       └─ NO (10+ % mismatch) →
  │   │           │
  │   │           ├─ File issue in Factory Core repo (proposing API extension)
  │   │           │
  │   │           └─ DECISION POINT:
  │   │               ├─ If extension accepted → Wait for implementation (1–2 sprints)
  │   │               └─ If extension rejected → You may build custom (isolation required)
  │   │
  │   └─ NO (not in Factory)
  │       │
  │       ├─ Is this feature:
  │       │   ├─ PLATFORM-LEVEL (auth, payments, storage, observability)?
  │       │   │   → **PROPOSE NEW FACTORY PACKAGE** (Phase C)
  │       │   │   → Meanwhile, build custom with clear seam for future swap
  │       │   │
  │       │   ├─ APP-SPECIFIC (business logic, workflows, domain models)?
  │       │   │   → **BUILD CUSTOM IN YOUR APP** (not Factory)
  │       │   │
  │       │   └─ UNCERTAIN?
  │       │       → Ask: "Will a different app need this in next 6 months?"
  │       │       → YES → **PROPOSE FACTORY PACKAGE**
  │       │       → NO → **BUILD IN YOUR APP**
  │       │
  │       └─ Assume you'll build custom...
  │           │
  │           └─ Do you have > 1 app that needs this?
  │               ├─ YES → Extract to Factory package after 2–3 apps prove the pattern
  │               └─ NO → Build inline; extract only if pattern becomes clear
```

## Common Mistakes to Avoid

### ❌ Don't:
1. **Import private exports** — Use only documented public exports (`index.ts`); package internals may break
2. **Bypass error handling** — Don't catch `FactoryBaseError` and re-throw as generic `Error`; consumers won't know how to handle
3. **Skip analytics instrumentation** — Without `.track()` calls, you have no visibility into user behavior
4. **Use Factory packages for business logic** — Packages are infrastructure only; domain models belong in your app
5. **Ignore Stripe webhooks** — Missing webhook handler = silent payment failures; use `stripeWebhookHandler` middleware
6. **Skip tenant context in multi-tenant scenarios** — Missing `setTenantId()` = cross-tenant data leaks

### ✅ Do:
1. **Always initialize monitoring first** — Sentry should be the first middleware
2. **Validate auth before accessing DB** — RBAC → DB query, never the reverse
3. **Log errors with context** — Include userId, tenantId, requestId for debugging
4. **Test package integration** — Use mock factories from `@latimer-woods-tech/testing`
5. **Keep packages platform-neutral** — No Node.js built-ins; use `TextEncoder` not `Buffer`
6. **Extend thoughtfully** — If a package doesn't fit, propose an extension before building custom

  ↓

         ↓        ↓

## Support & Extension

### How to Request a Package Extension
1. Open an issue in Factory Core: `@latimer-woods-tech/* package name: feature request`
2. Explain the gap and why it needs to be in Factory (not your app)
3. Link real app usage (e.g., videoking uses case)
4. Platform team reviews in weekly factory-core standup
5. If approved → estimated ship date + milestone

### How to Propose a New Package
1. Identify the use case (e.g., "we need compliance logging for GDPR")
2. Check if 2+ apps will consume it within 6 months
3. Propose in factory-core issue; include dependency map
4. Platform team adds to roadmap after Phase C stabilization
